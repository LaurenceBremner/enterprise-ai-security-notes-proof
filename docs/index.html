Scope

A compact review of what has to be true before an AI assistant enters a controlled pilot.

The artifact focuses on identity, tenant boundaries, evidence retrieval, auditability, retention, secrets, and rollout gates around a synthetic support-workflow assistant.

Trust boundaries

Where controls sit

  • IdentityOIDC claims map user, tenant, role, and admin scope before retrieval.
  • EvidenceConnectors enforce tenant and record-level filters before context reaches the model.
  • ActionHigh-risk drafts pass through eval scoring and human approval before send.

Data flow

Minimal path

1. User requestAuthenticated request carries tenant and role claims.
2. RetrievalAllowed evidence is filtered, redacted, and logged.
3. Model draftDraft response is marked with source references and confidence.
4. GateEval service blocks unsafe action or routes to human approval.
5. AuditDecision, evidence IDs, reviewer, and retention policy are recorded.

Risk register

Review questions

  • Cross-tenant leakProve connector filters cannot be bypassed by prompt or cached context.
  • Unapproved actionSeparate drafting from sending and require review on high-risk cases.
  • Audit gapCapture enough metadata to reproduce why a response was sent or blocked.

Methodology

How the notes were structured

  1. Start from a concrete assistant workflow and list the data it needs.
  2. Draw trust boundaries around identity, retrieval, model draft, action, and audit.
  3. Identify failure modes that would block a pilot.
  4. Map each failure to a control, owner, and verification question.
  5. Keep production caveats explicit without claiming a full security architecture.
